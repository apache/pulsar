Upgrade Guide

升级指南

Apache Pulsar 包含了 ZooKeeper、bookies 和 brokers 多种组件。 这些组件有些是有状态的，有些是无状态的。 除非有特殊需求，否则不必升级 ZooKeeper 节点。 升级时，需要注意 bookies(有状态的)、brokers 和 proxies (无状态的)。

以下是升级 Pulsar 集群的指南。 升级前阅读指南。

  • 在升级前备份所有配置文件。
  • 完整地阅读指南、制定计划、然后执行计划。 制定升级计划时，需要考虑到你的具体要求和环境。
  • 注意升级组件的命令。 一般来说，不需要升级 Zookeper 或者 配置存储集群 (全球 ZooKeeper 集群)。 首先需要升级 bookies，然后升级 brokers、proxies 和客户端。
  • 如果autorecovery启动，则需要在升级过程中禁止autorecovery，并且完成这个过程后重新启动。
  • 认真阅读每一个版本的注释。 版本注释包括可能影响升级的特征、配置变化。
  • 在升级集群中该类型的所有节点之前，升级每个类型的一小部分节点到 canary 测试新版本。 在升级 canary 节点时，先让其运行一段时间以确保其正确运转。
  • 如果集群以多集群复制模式运行，那么在升级所有的数据中心前先升级一个数据中心来验证新版本。

注意：目前 Apache Pulsar 兼容两种版本。

升级序列

升级 Apache Pulsar 集群时须遵循升级序列。

  1. 升级 ZooKeeper (可选择)。
  • Canary 测试：以一组或一小部分 ZooKeeper 节点测试升级版本。
  • 滚动升级：一次性将升级版本逐步部署到所有 ZooKeeper 服务器。 在整个滚动升级过程中监视仪表板。
  1. 升级bookies
  • canary 测试：以一组或一小部分 bookies 测试一个升级版本。 - Rolling upgrade: - a. 用以下命令禁用 autorecovery 。 shell bin/bookkeeper shell autorecovery -disable
    • b. Rollout the upgraded version to all bookies in the cluster after you determine that a version is safe after canary.
    • c. 升级所有的 bookies 之后，用以下命令重新启动 autorecovery。 
      bin/bookkeeper shell autorecovery -enable
  1. Upgrade brokers
  • Canary test: test an upgraded version in one or a small set of brokers.
  • Rolling upgrade: rollout the upgraded version to all brokers in the cluster after you determine that a version is safe after canary.
  1. Upgrade proxies
  • Canary test: test an upgraded version in one or a small set of proxies. - 滚动升级：确定 canary 之后的版本是安全的之后，将升级的版本推广到群集中的所有 proxies。

升级 ZooKeeper (可选择)。

升级 ZooKeeper 服务器时，可以先做 canary 测试，然后升级集群中所有的 ZooKeeper 服务器。

Canary 测试

升级集群中所有的 ZooKeeper 服务器之前，可以测试其中一个 ZooKeeper 服务器中的升级版本。

升级 ZooKeeper 服务器至新的版本需完成以下步骤：

  1. 停止 ZooKeeper 服务器。
  2. 升级二进制和配置文件。
  3. 用新的二进制文件启动 ZooKeeper 服务器。
  4. 用 pulsar zookeeper-shell 连接新升级的 ZooKeeper 服务器，然后运行新的命令来验证它是否如预期的那样运转。
  5. 运行 ZooKeeper 服务器几天，观察并且确保 ZooKeeper 集群运行的很好。

Canary 回转

如果在 canary 测试过程中出现了问题，那么可以关闭有问题的 ZooKeeper 节点，恢复二进制和配置，使用还原的二进制重新启动 ZooKeeper。

升级所有的 ZooKeeper 服务器

用 canary 测试升级集群中的一个 ZooKeeper 后，就可以升级集群中所有的 ZooKeeper 服务器。

可以根据以下 canary 测试步骤逐个升级所有的 ZooKeeper 服务器。

升级bookies

升级 bookies 时，可以先做 canary 测试，然后升级集群中所有的 bookies。 欲知更多的细节，可以阅读 Apache BookKeeper Upgrade guide。

Canary 测试

升级集群中所有的 bookies 之前，可以用一组或一小部分 bookies 来测试升级版本。

升级 bookies 至新的版本需完成以下步骤：

  1. 停止 bookies。
  2. 升级二进制和配置文件。
  3. 以 ReadOnly 模式启动bookie，以验证此新版本的bookie在读取工作量方面是否运行良好。 
    bin/pulsar bookie --readOnly
  4. 当 bookies 以 ReadOnly 模式成功运行时，停止 bookies 并以 Write/Read 模式重新启动。 
    bin/pulsar bookie
  5. 观察且确保集群服务器同时提供读写流量。

Canary 回转

如果在 canary 测试中出现了问题，那么可以关闭有问题的 bookies 节点。 其他集群中的 bookies 用 autorecovery 替换这个有问题的 bookies 节点。

升级所有的 bookies

用 canary 测试升级集群中的一些 bookies之后，就可以升级集群中所有的 bookies。

升级之前，必须决定是否一次升级整个集群，包括停机时间和滚动升级脚本。

在滚动升级脚本中，一次升级一个 bookies。 在停机时间升级脚本中，先关闭全部集群，升级每个 bookies，然后启动集群。

在两个脚本下进行升级时，每个Bookie的过程都相同。

  1. 停止 bookies。
  2. 升级软件 (新的二进制或是新的配置文件)。
  3. 启动 bookies。

Advanced operations When you upgrade a large BookKeeper cluster in a rolling upgrade scenario, upgrading one bookie at a time is slow. 如果你配置机架感知或区域感知放置策略，则可以按机架或按区域升级 bookies，从而加快了整个升级过程。

升级 brokers 和 proxies

brokers 和 proxies 的升级过程是一样的。 brokers 和 proxies 是 stateless，所以升级这两个脚本是很简单的。

Canary 测试

可以在升级集群中所有的节点之前用一组或一小部分节点测试升级版本。

升级新的版本需完成以下步骤：

  1. 停止 broker (或 proxy)。
  2. 升级二进制和配置文件。
  3. 启动 broker (或 proxy)。

Canary 回转

在 canary 测试的过程中出现问题时， 可以关闭有问题的 broker (或 proxy) 节点。 还原到旧版本，然后重新启动 broker (或 proxy) 。

升级所有的 bookies 或 proxies

在 canary 测试升级集群中的一些 brokers 或 proxies 之后，就可以升级集群中所有的 brokers 或 proxies。

升级之前，必须决定是否一次升级整个集群，包括停机时间和滚动升级脚本。

滚动升级脚本下，如果集群规模较小，则可以一次升级一个 broker 或 一个 proxy。 如果集群规模较大，则可以分批升级 brokers 或 proxies。 分批升级 brokers 或 proxies 时，确保集群中剩余 brokers 或 proxies 具有足够的容量来处理升级期间的流量。

停机时间升级脚本下，先关闭全部集群，升级每个 broker 或 proxy，然后启动集群。

在两个脚本下进行升级时，每个 broker 或 proxy 的过程都相同。

  1. 停止 broker 或 proxy。
  2. 升级软件 (新的二进制或是新的配置文件)。
  3. 启动 broker 或 proxy。
