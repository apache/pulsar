Pulsar 基本术语

下面是 Apache Pulsar 相关的一些术语:

概念

Pulsar

Pulsar 是一个分布式消息系统，最初由 Yahoo 创建，现在是 Apache 软件基金会的一个顶级项目。

消息（Message）

消息是 Pulsar 的基本单元。生产者（producer）将消息发布到 主题（topic），消费者（consumer） 从主题消费消息。

主题（Topic）

主题（topic）是传递消息的命名通道，主题将生产者（producer） 发布的消息传递给消费者（consumer），并由消费者来处理这些消息。

分区主题（Partitioned Topic）

分区主题有多个 Pulsar broker 提供服务，支持更高吞吐能力。

命名空间（Namespace）

相关 主题（topic） 间的分组机制。

命名空间 Bundle（Namespace Bundle）

同一个命名空间下的虚拟 topic 组。 命名空间 bundle 是 32 位的哈希值，取值范围从 0x00000000 到 0xffffffff。

租户（Tenant）

用于分配容量和执行身份验证/授权方案的管理单元。

订阅（Subscription）

由一组 消费者（consumer） 在 主题（topic） 上建立的消费关系。Pulsar 支持四种订阅模式（独占、共享、灾备和 key_shared）。

发布 - 订阅（Pub-Sub）

一种消息模式，即 生产者（producer） 进程发布消息到 主题（topic），然后 消费者（consumer） 进程消费（处理）这些消息。

生产者（Producer）

该进程将消息发送到 Pulsar 主题。

消费者（Consumer）

该进程建立了一个 Pulsar 主题订阅，并处理由生产者发布到主题上的消息。

Reader

消息处理程序，与 Pulsar 消费者（consumer） 非常相似，但二者有两个关键不同：

  • 可以指定 reader 在 topic 上开始处理消息的位置，而 consumer 总是从可用、未确认消息中的最新消息开始消费；
  • Reader 不保留数据，也不确认消息。

游标（Cursor）

消费者（Consumer） 的订阅位置。

消息确认（ack）

消费者发送给 Pulsar broker 确认消息，表明消息已成功处理。 Pulsar 通过 Ack 确认是否可以将消息从系统中删除。如果没有收到 Ack，则一直保留消息到处理完成。

否定确认（Negative Acknowledgment，nack）

当应用程序无法处理特定消息时，应用程序会向 Pulsar 发送 Nack，表示可以在一段时间后重放（replay）此消息。 (默认情况下，发送失败的消息会在一分钟之后被重新消费) 请注意，订单订阅类型为否定， 比如Exclusive，Failover和Key_Shared之类的消息可能会导致发送失败的消息以不符合原始顺序的方式到达使用者。

未确认（Unacknowledged）

一条消息已经发送给消费者来处理，但仍然没被消费者确认为已处理。

消息保留策略

在命名空间级别，通过设置消息大小和消息保留时间，为消息（已被 Ack）设置保留策略。

多租户

能够隔离 命名空间，指定配额，并在每个-租户的基础上配置认证和授权。

架构

单机（Standalone）

一个轻量级的 Pulsar broker，其中所有组件都运行在一个单一的 Java 虚拟机（JVM）进程中。单机集群可以在单台机器上运行，对于开发来说非常有用。

集群（Cluster）

一组 Pulsar broker 和 BookKeeper 服务器 (又称 bookies)。 集群可以部署在不同的地理区域，也能在 跨地域复制进程中复制消息。

实例（Instance）

作为一个整体运行的一组 Pulsar 集群。

跨地域复制

跨 Pulsar集群消息复制，主要是不同数据中心或地理区域。

配置存储（Configuration Store）

Pulsar's configuration store (previously known as configuration store) is a ZooKeeper quorum that is used for configuration-specific tasks. A multi-cluster Pulsar installation requires just one configuration store across all clusters.

Topic 查询

A service provided by Pulsar brokers that enables connecting clients to automatically determine which Pulsar cluster is responsible for a topic (and thus where message traffic for the topic needs to be routed).

服务发现（Service Discovery）

A mechanism provided by Pulsar that enables connecting clients to use just a single URL to interact with all the brokers in a cluster.

Broker

Pulsar 集群 的一个无状态组件，支撑运行另外两个组件：一个 HTTP 服务器，提供了用于管理和主题查找的 REST 接口，以及一个消息调度器（dispatcher），用于处理所有消息传输。 Pulsar 集群通常包括多个 broker。

调度器（Dispatcher）

一个异步 TCP 服务器，用于 Pulsar broker 上所有数据传输。Pulsar 调度器使用一个自定义的二进制协议进行所有通信。

存储

BookKeeper

Apache BookKeeper 是一个高可扩展、低延迟的持久化日志存储服务系统，Pulsar 使用它存储数据。

Bookie

Bookie 是单个 BookKeeper 服务器的名称，也就是 Pulsar 的存储服务器。

Ledger

Ledger 是 BookKeeper 的基本存储单元，用来持久化存储 Pulsar 主题中的消息。

Functions

Pulsar Functions 是一个轻量化的函数计算框架，可以从 Pulsar 主题消费消息，应用自定义的处理逻辑，也能将结果发布到主题上。
